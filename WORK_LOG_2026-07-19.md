## 1. What it is

You'd previously used a local Mongo install or Atlas. This replaces both with a
container you own, so all five projects come up the same way.

```
pedro_luis_imoveis_database/
  docker-compose.yml
  .env                     ← real values, gitignored
  init/01_create_app_user.js
  scripts/{backup,restore,migrate_from}.sh
```

## 2. Decisions worth knowing

**No `ports:` on the mongodb service.** Publishing 27017 would expose the
database to the entire internet on a VPS — this is the single most common way
self-hosted Mongo gets ransomwared. Only containers on `pedro-net` can reach it.
For local access use `docker compose exec mongodb mongosh`, or temporarily add a
`127.0.0.1:27017:27017` binding.

**Named volumes, not bind mounts** (`mongo_data`, `mongo_config`). The data
survives `docker compose down`, image upgrades and container rebuilds.

**Two users, not one.** The root user exists only to create the second;
`init/01_create_app_user.js` creates a least-privilege user scoped to the app
database, and that's the one in the backend's connection string. The init script
runs only on an empty volume — i.e. the very first start. **Changing it later has
no effect** unless you drop the volume.

**Healthcheck** with `start_period: 30s` so dependent services can wait on it
properly rather than racing the first connection.

## 3. Scripts

| Script | What it does |
|---|---|
| `scripts/backup.sh` | `mongodump` into a timestamped archive, prunes old ones |
| `scripts/restore.sh` | `mongorestore` from a chosen archive |
| `scripts/migrate_from.sh` | one-shot import from your existing local/Atlas database |

Run a backup on a cron on the VPS. Nothing here does that automatically yet —
that's a deliberate gap, since scheduling belongs to the host.

## 4. `.env`

Generated with real values during the session and is gitignored. It holds
`MONGO_ROOT_USER` / `MONGO_ROOT_PASSWORD` / `MONGO_APP_USER` /
`MONGO_APP_PASSWORD` / `MONGO_DATABASE`.

**If you lose it after the volume is initialised, the credentials in the volume
still apply** — the init script won't rerun to reset them. Keep a copy somewhere
safe.

## 5. Sizing — the 157GB question

- The listing documents are tiny: **~2.6KB each**. Even 100,000 listings is a few
  hundred megabytes. The database is not what fills a disk.
- **Images dominate.** At roughly 536KB per processed photo × 20 photos ≈ **11MB
  per listing**, so 157GB is roughly **10,000 listings** of headroom.
- That's ample. Revisit only if you start storing video at scale — the images
  service accepts up to 300MB per video file, and video will change the maths
  fast.

## 6. Switching between local, container and Atlas

Only the backend's `MONGO_DB` connection string changes. Nothing else in the
codebase knows where Mongo lives, so moving from this container to Atlas later
is a one-line change plus a `mongodump`/`mongorestore`.
